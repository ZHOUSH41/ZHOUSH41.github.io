Golang多线程的踩坑点

March 21, 2023 in code

问题：

从一个多线程处理并发问题谈起，首先有一个任务处理函数 DoTask(int) int，此函数根据输入计算一个结果并返回（任意的实现）。编写程序，用户可以输入n个需要处理的数据和最大的并行计算数 m。这n个输入数据通过DoTask函数并行的完成计算，每完成一次计算，就输出进度、已完成的计算结果的累加值。

// 任意的返回一个结果就行，这块随意实现
func DoTask(n int) int {
	return n*n
}

问题分析：

  1. 并发模式：首先这是一个多生产者单消费者的并发模式，那么可以生产者可以通过channel来发送数据
  2. 并发控制：问题中可以用条件变量或者channel来实现，最大并发数的控制
  3. 并发协调：因为需要等待所有生产者生产完数据之后才能退出，所以这里需要waitgroup来控制并发完之后的协程等待
  • 初版
func main() {
	n, m := 10, 5
	var wg sync.WaitGroup
	maxCh := make(chan int, m)
	for i := 1; i <= m; i++ {
		maxCh <- i
	}
	ch := make(chan int)
	for i := 1; i <= n; i++ {
		wg.Add(1)
		go func(n int) {
			fmt.Println("start go", n)
			<-maxCh
			ch <- DoTask(n)
			maxCh <- 1
		}(i)
		wg.Done()
	}
	totoal := 0
	sum := 0
// fEnd:
	for {
		select {
		case x := <-ch:
			totoal += x
			sum++
			fmt.Println(sum, totoal)
			if sum == n {
				// break fEnd
				return 
			}
		default:
			time.Sleep(time.Second)
		}
	}
	wg.Wait()
}

tips: for select中如果需要使用break，需要指定外层的for命名，不然只会跳过seclect 初版实现的问题：

  1. 虽然打印出来了对应的数据，但是没有控制并发的数量，因为实际上是进入了go func后在go func里做执行的时候阻塞住了，并没有控制到对应数量的
  2. 没有实现优雅退出，因为是通过消费端特定数量的实现来确定的退出机制的，这样实现不太优雅
  3. waitgroup实际上没有任何的用处，因为消费端是在主进程之中，并且是循环等待的，所以怎么都会等待所有生产者线程执行完

第二版实现：

func main() {
	ch := make(chan struct{}, 3)
	nCh := make(chan int, 3)
	var wg sync.WaitGroup
	go func() {
		idx := 0
		totoal := 0
		for i := range nCh {
			idx++
			totoal += i
			fmt.Println("receiver", idx, totoal)
		}
		fmt.Println("all done")
	}()
	// sender controller
	for i := 1; i <= 10; i++ {
		ch <- struct{}{}
		wg.Add(1)
		// worker
		go func(n int) {
			fmt.Println("start go", n)
			defer wg.Done()
			// do something
			nCh <- DoTask(n)
			time.Sleep(time.Second)
			<-ch
		}(i)
	}
	wg.Wait()
}

改动

  1. 把并发控制的channel前置到for循环当中，实际上就只有3个协程在工作了
  2. waitgroup可以做有效的等待了（但这里还是有问题） 问题
  3. 消费者线程打印的all done其实不会打印，因为主进程已经退出了
  4. 其实也没有实现优雅退出，因为并没有去关闭对应的channel

最终版：

func main() {
	ch := make(chan struct{}, 3)
	nCh := make(chan int, 3)
	var wg sync.WaitGroup
	var wg1 sync.WaitGroup
	wg.Add(2)
	// sender controller
	go func() {
		defer wg.Done()
		for i := 1; i <= 100000; i++ {
			ch <- struct{}{}
			wg1.Add(1)
			// worker
			go func(n int) {
				defer wg1.Done()
				// do something
				nCh <- DoTask(n)
				time.Sleep(time.Second)
				<-ch
			}(i)
		}
		wg1.Wait()
		close(nCh)
	}()

	// recevied controller
	go func() {
		defer wg.Done()
		idx := 0
		totoal := 0
		for i := range nCh {
			idx++
			totoal += i
			fmt.Println("receiver", idx, totoal)
		}
		fmt.Println("all done")
	}()
	wg.Wait()
}
  1. 主进程需要等待生产者和消费都结束之后才能退出，这个wg保证的
  2. 生产者在所有生产队列生产完毕之后close 发送的channel，实现了优雅的关闭逻辑

当然也可以使用，消费者去close一个channel，然后将这个消息proxy给生产者然后退出， 上面最终版的实现把多生产者再统一成一个管理者的模式，可以实现优雅的关闭，还可以拓展成多生产多消费的情况，而且代码易读性（易读性是个人的看法）比较好
